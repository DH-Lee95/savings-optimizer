# Architecture Decision Records

## 철학
MVP는 빠르게 신뢰 가능한 계산 경험을 제공하는 데 집중한다. 자동화보다 데이터 정확성과 계산 근거 노출을 우선하며, 금융상품 추천이 아니라 사용자 조건 기반 예상 계산 도구로 설계한다.

---

### ADR-001: 정적 MVP 우선, 유료 리포트부터 최소 서버 도입
**결정**: 초기 MVP는 의존성 없는 정적 HTML/CSS/ESM JavaScript로 구현했다. 유료 리포트 검증 단계에서는 Node HTTP 서버를 추가해 `/api/checkout`, `/api/reports`, `/api/reports/recalculate`에서 결제 사용권과 상세 리포트 생성을 검증한다. 실제 계정, PG 정산, 관리자, 스케줄러 운영이 확정되면 Next.js 15 App Router와 TypeScript strict mode로 전환한다.

**이유**: 초기 검증 단계에서는 설치 의존성을 줄이고, 모바일 계산 경험과 데이터 검수 흐름을 빠르게 확인하는 것이 중요하다. 현재 외부 데이터 처리는 Node.js 스크립트로 서버/운영자 환경에서 수행한다.

**트레이드오프**: 최소 Node 서버는 빠르게 유료 접근제어를 검증할 수 있지만, 운영 결제/계정/관리자 기능에는 한계가 있다. 공개 유료 서비스 전에는 PG 승인 콜백, 영수증/환불 처리, DB 저장소, 관리자 검수 화면을 붙여야 한다.

### ADR-005: 결제 사용권은 입력 조건 1세트에 묶는다
**결정**: 1회 결제는 결제 당시 입력 조건 해시에 묶고, 상세 리포트 생성 시 서버에서 같은 조건인지 검증한다. 리포트 안에서 추천 상품을 “이미 사용 중 또는 해당 조건 충족 안됨”으로 제외하는 재계산은 허용하되, 월 저축액/기간/우대조건 변경은 새 결제를 요구한다.

**이유**: 뒤로가기, 새로고침, 브라우저 콘솔 조작으로 여러 조건을 반복 분석하는 사용을 줄이고, 사용자가 결제한 리포트 범위를 명확히 하기 위해서다.

**트레이드오프**: 사용자가 결제 직후 조건을 잘못 입력했을 때 수정 UX가 빡빡할 수 있다. 운영 단계에서는 결제 후 짧은 유예 시간 또는 고객센터 재발급 정책을 별도 설계한다.

### ADR-002: 규칙 기반 최적화 우선
**결정**: MVP의 배분 로직은 선형계획법이나 고급 최적화 엔진 대신 규칙 기반 계산으로 시작한다.

**이유**: 금융상품 조건은 예외가 많고, 초기 검증 단계에서는 사용자가 계산 근거를 이해하는 것이 최적화 정교함보다 중요하다.

**트레이드오프**: 복잡한 목표함수와 다중 제약을 완전히 최적화하지 못할 수 있다. 이후 실제 사용자 입력과 상품 데이터가 쌓이면 최적화 엔진을 도입한다.

### ADR-003: 반자동 상품 데이터 관리
**결정**: 금융감독원 금융상품 한눈에 데이터와 주요 은행 상품의 수동 보정 데이터를 함께 사용한다.

**이유**: 우대금리, 가입 한도, 채널 조건은 자동 수집만으로 안정적으로 해석하기 어렵다.

**트레이드오프**: 운영자가 데이터를 검수해야 한다. 대신 잘못된 금리 노출과 깨진 가입 링크로 인한 신뢰 저하를 줄일 수 있다.

### ADR-004: 서버 측 외부 연동 제한
**결정**: 외부 금융 데이터 API, 크롤링 대상, 제휴 링크 생성은 클라이언트에서 직접 호출하지 않는다.

**이유**: API 키, 제휴 파라미터, 데이터 보정 규칙을 보호하고 응답 형식을 통제해야 한다.

**트레이드오프**: API 라우트와 서버 측 캐싱 계층이 필요하다. MVP에서는 단순한 서버 래퍼로 시작한다.
